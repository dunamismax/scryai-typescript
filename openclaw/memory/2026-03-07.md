## Discord bound-channel `/new` and `/reset` routing fix

- Bug: Discord native slash `/new` and `/reset` in the bound `#codex` channel could diverge from normal channel-message routing. The broken lane was `agent:main:discord:channel:1479615088053850253` with fallback label `discord:g-1479615088053850253`; the correct lane was `agent:codex-orchestrator:discord:channel:1479615088053850253` with label `discord:1479614326774956167#codex`.
- Root cause:
  - Native Discord command routing was still resolving with startup config instead of a fresh config snapshot, so late-added bindings could fall back to `main`.
  - More importantly, the native command path treated `configuredRoute?.boundSessionKey ?? ""` as a valid bound key. Because `""` is not nullish, `SessionKey` and `CommandTargetSessionKey` could be blanked instead of falling back to the resolved bound-agent session key.
- Fix:
  - Reload config inside the native Discord command route/model-picker path before resolving route/binding decisions.
  - Normalize empty bound-session results to `undefined` so native slash commands fall back to the resolved route session key instead of `""`.
  - Persist guild-channel metadata from the native path as `ConversationLabel = "Guild #channel channel id:<id>"`, `GroupChannel = "#channel"`, `GroupSubject = "#channel"`, and `GroupSpace = <guildId>` so the session store/display name stays `discord:<guildId>#<channel>`.
- Verification:
  - `pnpm vitest run src/discord/monitor/native-command.plugin-dispatch.test.ts src/routing/resolve-route.test.ts src/auto-reply/reply/session.test.ts`
  - `pnpm exec oxlint --type-aware src/discord/monitor/native-command.ts src/discord/monitor/native-command.plugin-dispatch.test.ts src/routing/resolve-route.test.ts src/auto-reply/reply/session.test.ts`
  - Restarted live gateway with `openclaw gateway restart`.
