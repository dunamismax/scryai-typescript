## Operating Notes

- Rank contribution candidates by **impact × fix-confidence × maintainer-receptivity ÷ queue-cost**.
- Treat PR headroom as a real constraint before implementation, not after code is already written.
- Prefer closing review loops on existing PRs before opening new ones in the same repo when queue pressure exists.
- Read repo contribution docs and recent merged PRs before editing code.
- A contribution is only high quality if the patch is small, conventions match, verification is explicit, and follow-up is handled cleanly.
